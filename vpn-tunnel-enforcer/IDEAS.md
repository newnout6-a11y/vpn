# Идеи и бэклог для будущих доработок

## 1. Продвинутый мониторинг трафика (Process-Level Traffic Monitor)

**Проблема:** 
Текущая вкладка «История трафика» берет данные из логов `sing-box`. Это позволяет показывать только домен и количество подключений, но не позволяет определить, **какое именно приложение** (имя процесса, например `chrome.exe`, `telegram.exe`) инициировало запрос. VPN-адаптер (TUN) на уровне Windows видит только голые IP-пакеты.

**Предлагаемое решение:**
Превратить страницу трафика в полноценный локальный файрвол (наподобие GlassWire), который будет показывать имена процессов.

**Архитектурные изменения:**
1. **Перевод ETW-модуля (vpnte-etw-sidecar) в постоянный режим работы.**
   Сейчас sidecar запускается только для «Глубокого анализа» и сбора диагностического архива. Его нужно будет перевести в режим фоновой службы (always-on).
2. **Трекинг PID -> Process Name.**
   Модуль на Rust должен будет не только ловить сетевые пакеты через ядро Windows (ETW/WFP), но и резолвить `ProcessId` (PID) в имена исполняемых файлов (`ProcessName`), кэшируя их, чтобы не нагружать систему.
3. **Корреляция потоков данных (Data Joining).**
   Реализовать механизм слияния данных:
   - Данные о DNS-запросах из `sing-box.log` (чтобы знать домен).
   - Данные об установке TCP/UDP соединений из ETW-драйвера (чтобы знать PID и Process Name).
4. **Связь IPC с UI.**
   Стримить эту объединённую информацию в Electron-приложение (на вкладку «Трафик»).

**Риски и компромиссы:**
- **Расход батареи и CPU:** Постоянный анализ сетевого трафика на уровне ядра через ETW будет потреблять немного больше ресурсов процессора.
- **Объём логов:** `events.ndjson` от сайдкара может достигать больших размеров, потребуется реализовать агрегацию прямо в памяти или ротацию.
- **Сложность разработки:** Высокая. Потребует сложной синхронизации асинхронных событий (DNS-ответ пришел в одну секунду, а TCP-соединение от процесса открылось через 100мс).
